✍️ 블로그 글쓰기 도우미 개선 — BW-1~5 결과 보고

2026-07-08 · 운영자 피드백 5건 → 요구사항 인덱싱 → 반영 완료 · 대상: .github/workflows/nb-blog.yml(실사용 생성 경로) + index.html(nb-* 도구) + src/index.js(프롬프트 동기)

1. 통합 요구사항

블로그 글쓰기 도우미의 초안이 ① AI·기사 냄새 없는 사람 말투로(BW-1), ② 공연·출연진·작품 리서치로 살이 붙은 채(BW-2), ③ 이전 초안을 잃지 않고 오가며 다듬을 수 있게(BW-3), ④ 네이버 발행용 태그 30개(여수·순천·광양 검색 유입 포함)까지 한 번에(BW-5) 나오게 한다. 상세페이지 분석 흐름(BW-4)은 호평이므로 그대로 두고 그 위에 얹는다.

2. 인덱스별 피드백 → 원인 → 해결

인덱스피드백 요지원인 (실측)반영한 해결책
BW-1말투가 AI·기사 톤 — 예: 「이 뜻깊은 해를 기념해, 피아니스트 조재혁이 예울마루 대극장 무대에 오릅니다.」생성 프롬프트의 문체 지시가 「자연스러운 블로그 문체」 한 줄뿐 — 금지 패턴·예시 부재문체 계약 주입: 보도자료·AI 상투구 금지 목록 + 운영자 실례를 그대로 나쁜 예로 쓴 before/after + 사람 말투 원칙(문장 리듬·구체 사실·담당자 시점) + 출력 전 셀프 퇴고 지시. 러너에서 claude -p의 작업 폴더를 격리해 레포 개발 지침(CLAUDE.md)이 글쓰기 컨텍스트에 섞이던 것도 차단
BW-2공연·작가·곡·연주자 리서치 기능 — 글에 살이 없고, 손 리서치는 시간이 너무 듦생성이 claude -p --max-turns 1 = 도구 사용 불가, OCR 텍스트만으로 작성자동 웹 리서치: 새 초안 생성 시 WebSearch·WebFetch 허용(멀티턴 20) → 출연진 이력·곡/작품 배경·화제성을 3~6회 검색해 확인된 사실 2~4개를 본문에 녹임. 확인 못 하면 안 씀(추측 금지), 일시·가격 등 기본 정보는 OCR이 항상 정본. 조사 요약은 🔎 리서치 노트(접이식)로 함께 반환 — 글쓴이 팩트체크용
BW-31차 초안으로 되돌아가 읽고, 양쪽에서 좋은 문장을 골라 쓰고 싶음output 단일 문자열이라 재생성 시 이전 초안이 덮여 소멸버전 스냅샷: 생성·코멘트 반영마다 1차·2차·3차… 칩 생성, 클릭으로 왕복. 떠나는 버전에 직접 편집분도 저장돼 유실 없음. 저장은 인메모리 — 공용 PC 원칙(창 닫으면 소멸) 그대로 유지
BW-4상세페이지 읽어주는 기능 최고 (칭찬)—현행 유지 — OCR→분석 흐름 무변경. BW-2 리서치가 이 OCR 사실 위에 얹히는 구조(OCR = 정본, 검색 = 살)
BW-5네이버 태그 30개 추천 — 여순광 사람들 검색에 뜬금없어도 걸릴 태그 분석까지태그 기능 자체가 없음(해시태그 금지 지시만 존재)태그 30개 동시 산출: 4축 배분(공연 고유 8~12 / 예울마루·기관 4~6 / 여순광 생활·여행 유입 8~10 / 타겟·시기 4~6), 실검색어 형태·중복 금지. 결과 아래 칩으로 표시 + [태그 전체 복사] = 「#태그 #태그 …」 한 줄. 코멘트 수정 시 수정된 본문 기준으로 재산출

3. 무엇이 어떻게 바뀌나 (전/후)

말투 (BW-1) 전 — 「이 뜻깊은 해를 기념해, 피아니스트 조재혁이 예울마루 대극장 무대에 오릅니다.」 (기사 톤 — 프롬프트가 이런 문장을 막지 않았음) 후 — 프롬프트가 위 문장을 그대로 나쁜 예로 박고, 「라흐마니노프가 태어난 지 올해로 150년이에요. 조재혁 피아니스트가 그 기념 프로그램을 들고 여수까지 옵니다.」 방향을 좋은 예로 제시. 「~를 선사합니다」 「~의 향연」류 금지 목록 + 셀프 퇴고까지 강제 내용 밀도 (BW-2) 전 — 상세페이지 OCR에 적힌 것만으로 작성 → 공연만 띡 홍보하고 끝, 살이 없음 후 — 생성 전에 출연진·작품·화제성을 자동 검색해 확인된 이야기 2~4개를 녹임 + 리서치 노트로 근거 제시(몇 분 걸리는 대신 손 리서치 시간 대체) 작업 흐름 (BW-3·BW-5) 전 — 코멘트 반영하면 이전 초안 소멸(복사해 딴 데 저장해야 함) · 태그는 매번 손으로 고민 후 — 1차·2차… 칩으로 왕복하며 양쪽에서 문장 채집 · 태그 30개 자동 추천 + 원클릭 복사

4. 새 결과 화면 (헤드리스 실측 스크린샷)

결과 초안 아래로 💬 코멘트로 다듬기 → 🏷️ 추천 태그(+전체 복사) → 🔎 리서치 노트(접이식) 순. 초안이 2개부터는 결과 위에 1차·2차 버전 칩이 생긴다. 전부 기존 nb-* 정본 컴포넌트·토큰 계승(신규 raw hex 0).

6. 리스크·한계 (정직 표기)

  • 시간 리서치 포함 새 초안은 보통 2~5분(러너 기동+검색+집필). 화면에 안내 문구 표시, 폴링 한도 10분(동시 요청은 한 줄 큐라 순서가 밀리면 안내 후 재시도). 코멘트 수정은 리서치 생략이라 기존 속도.
  • 형식 모델이 ===TAGS=== 형식을 어기면 태그 없이 본문만 온다(파서가 안전 폴백 — 깨지진 않음). 태그가 안 오면 태그 박스가 그냥 숨는다.
  • 검색 실패 검색이 소득 없으면 종전처럼 주어진 정보만으로 작성하도록 지시(추측 금지 유지). 리서치 노트의 사실도 발행 전 한 번 훑기 권장 — 그래서 출처를 같이 적게 했다.
  • 동시 요청 생성은 한 줄 큐(concurrency) — 두 명이 동시에 누르면 뒷사람은 앞 작업 종료 후 시작. 리서치로 개별 작업이 길어진 만큼 대기가 생길 수 있음(job timeout 20분으로 교착 방지).
  • 프라이버시 버전 히스토리는 인메모리 — 창을 닫으면 소멸(공용 PC에서 다음 사용자에게 초안 노출 차단 원칙 그대로).
  • 디자인 신규 UI 전부 기존 nb-* 컴포넌트·토큰 계승 — check_design.py 1641/1641 동결 유지(신규 raw hex 0).
