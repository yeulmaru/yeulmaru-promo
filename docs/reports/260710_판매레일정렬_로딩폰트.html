2026-07-10 (KST) · index.html · 홈 화면 좌 캘린더 ↔ 우 「판매 현황」 레일 상단 정렬

1증상

데이터 로딩 중에만 우측 「판매 현황」 헤더가 좌측 요일바보다 아래로 어긋남. 로딩이 끝나면 정상. 운영자 신고 화면 그대로.

2정렬 구조 & 원인 (실측 확정 — 추측 아님)

좌우 두 열은 각각 별도 서브트리(중앙 #main-area / 우측 #sales-rail)라, 정렬을 _srailAlignTop()이 JS로 맞춘다: 우 스페이서 #srail-ttlzone 높이 = 좌 헤더밴드 .hdr 높이. 두 열은 같은 flex 행이라 top이 같고(railTop==mainTop), 정렬은 이 한 방정식으로 환원된다:

GAP = ttlzoneH − hdrH (0 이어야 정렬 · >0 이면 우측이 아래로)

그런데 로딩 중엔 _srailInit이 이 정렬을 딱 1회만 호출한다. 그 직후 비동기 웹폰트(Noto Sans KR / Pretendard)가 지연 로드되며 .month-text(32px) 라인박스가 바뀌어 .hdr 높이가 리플로우되는데 — 데이터 로드(_srailRender) 전까지 재정렬 훅이 없어 ttlzone이 옛 높이로 굳는다.

부트 → _srailInit 1회 정렬(폴백폰트 기준, hdr 큼) → 웹폰트 도착 → hdr 축소 재정렬 훅 공백 ttlzone stale → 데이터 로드 _srailRender 재정렬(정상 복귀)

사용자가 본 「우측이 아래」 방향 = 사용자 OS 폴백폰트(예: Malgun Gothic / Apple SD Gothic)의 32px 라인박스가 Noto보다 커서 정렬시점 hdr가 더 컸다는 뜻과 정확히 일치.

3재현 & 검증 (헤드리스 통제실험)

헤드리스 크로미움으로 로딩 상태를 고정하고, 폰트 스왑(폴백→웹폰트)에 의한 hdr 높이 변화를 모사해 관측자 ON/OFF 대조로 인과를 확정:

단계 (hdr 변화)관측자 ON 수정관측자 OFF 대조
기준선GAP 1pxGAP 1px
폴백: hdr 66→92 확대GAP 1pxGAP 1px
웹폰트: hdr 92→66 축소GAP 1px ✓GAP 25px ✗ (버그)

즉 관측자를 끄면 딱 사용자 증상(GAP 25px)이 복귀 — 이 관측자가 인과적 수정임을 증명. PAGEERROR 0.

4수정

_srailArmResize(1회 장전, _srailInit이 호출)에 순수 레이아웃 훅 2개 추가:

  • ResizeObserver를 .hdr·요일바에 걸어 크기가 변할 때마다 즉시 _srailAlignTop 재실행(자기교정) — 폰트·줌·칩줄바꿈·리사이즈 등 모든 원인 커버.
  • document.fonts.ready 재정렬 훅(폰트 로드 완료 시점 명시 · 방어적 이중화).

디자인 토큰·색·CSS 값 창작 0 → 게이트 check_design 1635/1635 불변. _srailAlignTop은 요일바 sticky top(위치)·우측열 크기만 조정 — 관측 대상의 box 크기는 안 바꿔 되먹임 무한루프 불가. 미지원 구형은 기존 render/resize 정렬로 폴백(회귀 0).

5적대 검증 (§분신술 5인 · 만장일치)

최상위 모델 READ-ONLY 5인 병렬 감사 — 전원 APPROVE:

  • 되먹임/성능: 관측대상 크기를 콜백이 안 바꿔 무한루프·스래싱 구조적 불가.
  • 부트/엣지: 유휴잠금 재진입도 armed 가드+영속노드로 단일 유효 관측자(누수·중복 0). 진입 애니(ae-pre/ae-in)는 transform/opacity라 무영향.
  • 호환/누수: 데스크톱 크롬 100% 지원, 관측 노드는 페이지 수명 싱글턴이라 disconnect 불요(기존 관측자 3건과 동일 하우스 패턴).
  • 근본원인: 폰트 스왑 리플로우 진단 end-to-end 정합, 완료상태 무해, 방향 무관(관측자가 어느 방향이든 교정).
  • 기틀/품질: 기틀 위반 0, 기존 measure-and-set 패턴 정확 계승, 최소침습.

지적된 LOW(주석 문구 정밀화 · 빈 catch 진단로그)는 반영 완료.

6리스크 & 잔여

  • 수용: 폰트 로드 시 ResizeObserver와 fonts.ready가 둘 다 _srailAlignTop을 부를 수 있으나 멱등(절대 px 대입)이라 무해 — 의도된 방어적 이중화.
  • 범위 밖(별개 · 기존 설계): 로딩 중 스크롤하면 sticky인 .hdr·요일바가 붙고 비-sticky 레일은 흘러 별도 어긋남 — 이번 버그(스크롤 top 로딩)와 무관한 기존 동작. 레일을 sticky로 바꾸는 건 구조 변경이라 운영자 판단 대기.
