## 🚨 제1 절대명령 — 디자인은 「루트 디자인 토큰」 계승 or 갱신, 둘 뿐 (레포 최상위 불변, 260704)

> ⚠️⚠️ 이 레포의 모든 디자인 변경(색·크기·간격·반지름·blur·모션·그림자·타이포·z·버튼·컴포넌트)은 예외 없이 아래 두 갈래 중 하나여야 한다. **세 번째 길은 없다.**
> - **① 계승(inherit) = 디폴트** — 필요한 값이 루트 디자인 토큰(**값 SSOT = `index.html`의 `:root` 2블록**)에 이미 있으면 그 토큰을 `var()`로 참조. **raw 색값 창작 절대 금지**(단, 같은 토큰의 alpha 변주는 정상 — 기틀 §3.5②). 구조값(radius·z·셀 등)도 **토큰이 있는 축이면 토큰 강제**(토큰 축이 없는 일반 px 레이아웃값은 기존 패턴 계승).
> - **② 갱신(update)** — `:root`에 없거나 기존 값을 바꿔야 하면 루트 토큰을 갱신. ⚠️ 갱신은 반드시 운영자에게 **먼저 묻고 승인**받은 뒤에만. **갱신의 특수형 = 운영자 승인 리터럴**: `var()`가 물리적으로 불가능한 위치(HTML `theme-color` 메타, baked PNG 등 빌드 에셋 색)는 승인 후 리터럴로 넣고 baseline/기틀에 사유 등재(§5 이력 전례) — 승인 없인 이 경로도 위반.
>
> **판정 절차(강제)**: 1) 필요한 값이 `:root`에 있나? → 있으면 계승(`var()`). 끝. 2) 없으면 → **멈추고 운영자에 묻는다**: 『기틀에 없다 — 갱신할지, 무엇을 계승할지(가장 가까운 기존 토큰 후보 제시)』. 임의 raw 박기 = 제1명령 위반. 3) 갱신 승인 → 즉시 편입 3점 세트: `:root` 토큰 추가/수정 → `docs/디자인기틀.md` 등재 → `tools/check_design.py` baseline 갱신+**사유 주석**.
>
> **색은 팔레트 폐쇄형** — `docs/디자인기틀.md` **§0 브랜드 컬러 확립본**의 컬러칩이 전부다(코어 인디고 `--accent` · 서브코어 살몬 `--c2` · 카테고리 6색 `--c1~--c6` · 역할색 각 딱 하나 · 중립 사다리). 칩 밖 색 반입·유사색 즉석 창작 = 위반. 시각본: `docs/브랜드컬러_확립본.html`.
>
> **왜**: raw 산개 = 하나 고칠 때 나머지 안 따라옴 = 품질 복리 붕괴 + 운영자 스트레스. 단일 토큰 SSOT + 계승/승인갱신만이 일관성 보장. **강제(기계)**: `tools/check_design.py`(baseline 래칫) + 편집 직후 훅(`design_gate.py`) + 커밋 게이트(`.githooks/pre-commit`) + 세션 계약 주입(`design_digest.py`).
>
> **📌 규칙 2 = 정본 위치 인덱스 — 규칙 1은 규칙 2를 절대적으로 따른다 (260704)**: 값·규칙·컴포넌트·확립본·강제 게이트가 **어디에 있는지**는 `docs/절대명령2_정본인덱스.md`(규칙 2)가 위치 SSOT — 새 정본 문서·확립본은 **그 표에 등재해야 정본**이다(미등재 = 참고 자료일 뿐). 단 내용 위상은 불변: 값 SSOT=`index.html :root`, 규칙 내용 SSOT=`docs/디자인기틀.md`(규칙 2는 위치만 정본).

- c) 도메인(데이터 흐름·인증·시트·함정·환경) 정본 = `docs/앱지침.md`.

## 2. 일하는 방식 (공통 · 항상 로드)

### 1) §운영 태도
- a) **한국어·반말.** 응답은 첫 글자부터 끝까지 한국어(분석·설명·도구 설명 전부). 예외 = 코드·식별자·경로·고유명사 원문 토막만. **커밋 메시지·코드 주석도 한국어 유지**(사용자가 영어로 요청한 경우만 예외). 반말이라고 대충 금지 — **풀어서 + 프로세스로**(왜 이런지 → 뭐부터 → 그다음) 차근차근.
- b) 사족·아부 빼고 **결론부터**. *내가 틀렸을 수 있다*가 기본값 — **주장 전에 검증**(파일·실행 확인·근거 제시), 추측을 사실처럼 말하지 않는다. 측정으로 뒤집히면 변명 없이 정정. 사용자 전제도 틀려 보이면 근거로 반박.
- c) **추측 패치 금지(이 프로젝트 제1원칙).** 브라우저 콘솔은 Claude가 못 본다 — 원인 모르면 진단 로그 박고 → 사용자가 시크릿창 캡처 → 원인 확정 → 수정.
- d) **혼자 맴돌지 마라.** 같은 시도 2번 실패 = 즉시 멈추고 질문(환경·경로·계정은 사용자가 1초면 안다).
- e) **능동 제안 = 디폴트.** 분기·교착 끝에 `한 수` 1개(없으면 *없음*). 의도 안 좁혀지면 *이거 맞아?*로 목표부터 확정.
- f) **정직 표식**: 추측·제안엔 OK확인/추정/제안 딱지 필수. `됐다 거짓`은 단정에만 걸림.

### 2) §작업표준 — 작업 4대 표준 (매 작업 실행방식 최상위 · 라우팅 판정엔 종속)
**실질 변경의 정의** = 운영자가 의도해 동작·구조·출력이 바뀌는 것. 조회·진단·오타·순수 표기 미세수정 = 실질 변경 아님 → b)·c) 면제. 작업 대상 라이브 = https://yeulmaru.github.io/yeulmaru-promo/ (main 머지 후 1~2분 빌드 + **브라우저 캐시버스트**(`Ctrl+Shift+R` 또는 `?cb=난수`) 필요 — `docs/앱지침.md` §환경 제약).
- a) **의도 명확화 = 착수 전 필수 · `왜`부터 3단**: {왜 시켰지?(목적) → 방향? → 최선 방법?}으로 근본·전체를 해결. 표면 한 줄 땜질 금지. 불분명하면 되짚어 재확정 후 착수(분기면 §선택질문). 착수 시 `git fetch origin main` + `git status`로 어느 커밋 위인지부터 확인. 단 `git` 입력이면 라우팅 0순위 선행.
- b) **결과 제시 = 쉬운 설명 + HTML 보고서**: 실질 변경은 전/후 비교·장단점·리스크를 담아 `docs/reports/`의 {yymmdd}_{라벨}.html 단일파일(인라인 CSS)로. *작은 변경은 채팅 마크다운 표로 갈음 가능.* (결정 **전** 디자인 제안·시안·튜닝 = §🎛 별도 정본 — 정적 보고서와 다른 산출물.)
- c) **검증 = §검증 표준대로 후 머지**: 특히 **데이터 변경·크로스유저·비동기·로그인/보안이 얽힌 수정 = §분신술 필수**(이 레포 실증 — 단독 검증으론 HIGH 회귀를 못 잡았다). `index.html` 수정 시 커밋 전 `node --check` 문법 검사(3 스크립트 블록) 관행 유지. **커밋은 작게, 한 변경마다 즉시 검증** — *전부 새로 설계하자* 충동이 와도 원인만 특정되면 대개 10줄 안쪽으로 끝난다.
- d) **누적 = 원장 append**: 완료 작업은 `docs/작업이력.md`에 append-only 누적(새 블록은 맨 위·기존 블록 수정 금지), 개선점은 같은 문서 §우선사항에.

### 3) §선택질문 — 선택 분기 = 버튼
- a) 사용자 선택 분기는 텍스트 나열 말고 AskUserQuestion 버튼. 맥락 한 줄만 위에, 자유 입력은 Other.

### 4) §PR웹훅 — PR 구독 = 머지 즉시 해제 (컨텍스트 보호)
- a) *watch/지켜봐* 명시 없으면 자동 감시 안 함. **PR 머지 직후 unsubscribe 즉시 호출**(재구독되면 또 해제). webhook 블록은 답장 없이 처리.

### 5) §모델
- a) 세션 작업자·검증가 = 최상위 모델·최고 노력. 검증·평의회도 동급 유지. 하향은 운영자 승인 후에만.

### 6) §UI계승
- a) **새 화면·컴포넌트를 0에서 만들지 마라** — `index.html`의 기존 패턴(정본 컴포넌트·토큰)을 최대 재사용, 톤 튀는 디자인 금지. 확신 없으면 비슷한 기존 요소를 그대로 따라 할 것. 상세 = 위 §🎨 + `docs/디자인기틀.md`.

## 3. 저장소·백업 원칙 + 수정 모드 (§수정 모드 · 운영 고정)

### 1) 원칙
- a) **단일 원본 = GitHub `yeulmaru`/`yeulmaru-promo`(main).** 라이브 = GitHub Pages(main에서 빌드). ⚠️ **Worker(`src/index.js`)는 예외** — git 반영 ≠ 배포, Cloudflare에 별도 배포해야 반영(`docs/앱지침.md` §시스템 설계).
- b) **수정 모드 트리거 = `git`(대소문자 무관) 또는 `깃`** — 즉시 시스템 파일 편집 모드.
- c) **롤백** = *그 버전으로* 한마디 → `_versions/`에서 복원(앱 코드는 git 이력로도 가능).

### 2) 수정 모드 절차 (반드시 이 순서)
- a) 편집 전 `git fetch origin main` → 최신 main 기준으로 시작. **이 레포는 여러 세션 동시 편집** — 옛 base로 머지하면 남의 변경이 되돌아간다(필요 시 rebase).
- b) 기틀 문서(라우터·지침·게이트)를 고칠 땐 현재본을 `_versions/`의 {yymmdd_HHmm}_{라벨}/ 폴더에 **백업**(구조 변경 시 필수 — 오타 등 국소 수정은 git 이력 갈음 가능, §기틀 보호 판정 기준). 앱 코드는 git 이력으로 갈음 가능.
- c) 수정. (**기틀 변경이면 수정 전에 운영자 확인** = §기틀 보호.)
- d) 게이트 통과 확인 — `python3 tools/check_design.py`(`index.html`·signage 편집 시) + `python3 tools/check_refs.py`(라우터·지침 편집 시) → `git add·commit·push`(작업 브랜치).
- e) **PR 생성 → §검증 통과 후 즉시 main 자동 머지**(draft 방치 금지 = 라이브 확정). **검증만 통과하면 머지 재확인 안 물음 = 자동 머지가 디폴트**(이 레포 = 실시간 운영 대상 아님 → 운영자 승인 대기로 draft 방치 금지) · 운영자에겐 **사후 요지 통지**로 갈음(사전 승인 대기 불요) · 단 §기틀·데이터·크로스유저·보안 얽힘은 통지에 핵심 요지 명시. **main force-push 금지**(작업 브랜치 force-push·reset은 이 자동 흐름서 허용) · 이미 머지된 PR 브랜치에 새 커밋 금지(최신 main에서 새 브랜치).
- f) `git fetch` 후 **`git show origin/main:<파일>`로 원격 실내용 직접 읽어 검증·보고**(화면·로컬만 보고 `됐다` 금지). 라이브 화면 확인이 필요한 변경은 Pages 빌드 후 캐시버스트로.
- g) 머지 충돌: 일반 파일은 `git merge origin/main -X ours` 흡수 가능 · 진짜 충돌 = STOP+보고 · `force` 안 씀. 단 **append-only 원장(`docs/작업이력.md`)은 `-X ours` 금지** = 수동 정리로 양쪽 보존.

### 3) 반영 보장 (불변·최우선)
- a) 변경 요청 = main 반영 + 실측 검증까지가 `완료`. 막히면 우회로 끝까지 반영, 그래도 안 되면 **`됐다` 절대 금지 → STOP+보고**. *검증 안 된 완료 보고는 거짓 — 모든 편의·속도보다 우선.*

### 4) 동시 세션 동기화
- a) main = 단일 진실원본. 각 수정은 그때그때 main에 머지(브랜치 방치 = 새 세션이 옛 룰을 봄). 열려 있던 세션 = 옛 복제본 → `git fetch` 재동기화. 큰 편집 전 *지금 이 파일 딴 데서 만지는 중?* 한 번 확인하면 충돌이 크게 준다.

## 4. 기틀 보호·검증

### 1) §기틀 보호 — 구조 변경은 반드시 사용자에게 묻는다
- a) **대상(기틀)**: 라우터 구조 · 지침↔라우터 분리 · 디자인 토큰 체계(§🚨) · 게이트 스크립트·훅 구조 · 출력 포맷 골격 · 공용 불변. **스스로 결정 금지, 운영자 승인 후에만.**
- b) 왜: 모델은 `효율화` 명목으로 구조를 손대고, 그 조용한 누락이 품질을 복리로 갉는다. 주변부(문구·임계값)는 자유 — 기틀만은 반드시 확인 후.
- c) **판정 기준**: 기틀 문서의 오타·깨진 참조·순수 표기 수정은 구조 변경이 **아니다** → 자가검증으로 충분(백업도 git 이력 갈음 가능). 절 신설·삭제·판정 규칙·게이트 로직 변경 = 구조 변경 → 운영자 확인 + §기틀검증 발동.

### 2) §기틀검증 — 기틀 변경 = 최상위 모델 5명 교차검증 필수
- a) 절차: §기틀 보호 사용자 확인 먼저 → 확정 전 독립 5명 병렬 발사(적대적 감사·레포 실측) → **전원 APPROVE 아니면 STOP·재설계** → 통과 후에만 §수정 모드 절차 전체로 확정.

### 3) §검증 — 다앵글 적대적 만장일치
- a) 모델·노력: 검증가 전원 최상위 모델·최고 노력.
- b) 인원 티어: 가) 상시 = 자가검증(게이트 + `node --check` + read-back) 나) 승격 = 명시 요청 또는 기틀 또는 실질 코드·구조 변경 또는 데이터·크로스유저·비동기·보안 얽힘 = §분신술 다) 국소 손질·순수 문구 = 자가검증 라) 감축은 사용자 승인 후에만 마) 순수 시각·목업 = 면제(시안은 HTML).
- c) 각기 다른 앵글로 적대적 감사 · 추측 금지 = 실측 근거.
- d) **READ-ONLY 하드 계약**: 검증 에이전트는 분석·보고만 — 레포 1바이트도 못 바꿈(git 조작·파일 생성/편집·PR 전면 금지 · 산출 = 텍스트만 · **메인이 레포 변경의 유일 행위자**). 강제 3중 = 가) 소환 프롬프트 첫 줄에 계약 명시 나) 읽기전용 에이전트 타입만 사용 다) 발사 후 rogue 브랜치·커밋 사후점검.
- e) **묘사 말고 실제 소환**: 반드시 실제 에이전트 툴로 N명 병렬 발사(머릿속 대체 금지).
- f) **만장일치 루프**: 발사 → 평결 종합 → 문제 전부 메인이 직접 수정 → 재검증 → 전원 OK까지 루프 → read-back 실측 → 그제서야 완료 보고.

### 4) §분신술 — 다인 평의회 호출어
- a) 운영자가 `분신술` 하면(또는 §검증 b-나 승격 조건 충족) → 최상위 모델 **기본 5명**(운영자가 인원 명시하면 그대로, 풀 호출 = 10명) 실제 병렬 소환(READ-ONLY 계약). 각자 다른 앵글(피드백·검증·회귀·효율·품질·아이데이션)로 → 수렴 평결 → 메인 조치 → 베스트픽.
